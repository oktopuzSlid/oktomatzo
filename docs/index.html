Oktomatzo Host Documentation

A full-stack platform that hosts independent interactive projects — 3D visualizers, minigames, AI tools, or anything else — under a single dashboard.

What Is This?

Oktomatzo Host is a platform that wraps independent projects with shared services: authentication, data persistence (save/load), leaderboards, and file storage.

Each project is a self-contained folder in projects/{slug}/. It works fully on its own — open the folder's index.html directly in a browser and it runs. When served through the platform, the project is wrapped with navigation, header, footer, and a JavaScript SDK that provides access to the platform's API.

A project doesn't need to know about the platform to work. The SDK is optional. If your project doesn't need auth or saves, just drop it in and it appears.

Key Principles

  • Projects are folders. Each project lives in its own directory under projects/. No Astro pages, no config files needed.
  • Zero build to add. Create a folder with the right files, and the platform picks it up automatically. No rebuild, no restart.
  • The SDK respects independence. window.Platform is available if the project wants it. A project that doesn't use it runs exactly the same.
  • Same origin. The platform and projects are served from the same domain. No CORS, no iframe complications.

Adding a New Project — Step by Step

Adding a project takes 4 steps. The project works immediately without any build step or server restart.

What Goes in the Folder

projects/my-minigame/
├── project.json     ← REQUIRED: metadata (title, description, icon, category)
├── index.html       ← RECOMMENDED: standalone HTML (works when opened directly)
└── main.js          ← RECOMMENDED: your game logic (loaded by the platform wrapper)
Only project.json is strictly required. The platform wrapper loads main.js automatically. Without index.html, the project only works through the platform URL, not standalone.

1 Create the folder

projects/my-minigame/

mkdir projects\my-minigame

The folder name becomes the URL slug: /projects/my-minigame/

2 Add project metadata

projects/my-minigame/project.json

{
  "slug": "my-minigame",
  "title": "My Minigame",
  "description": "A fun browser game with score saving.",
  "category": "interactive",
  "icon": "smile"
}

Categories: threejs, data-viz, interactive, tool.
Icons: smile, cube.

3 Write the project HTML

projects/my-minigame/index.html

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>My Minigame — Oktomatzo Host</title>
  <link rel="stylesheet" href="/css/style.css">
  <style>
    body { background: #000; min-height: 100vh; display: flex;
           align-items: center; justify-content: center; }
    #project-root { text-align: center; }
  </style>
</head>
<body>
  <div id="project-root">
    <p style="color:var(--text-muted);">Loading...</p>
  </div>
  <script src="main.js"></script>
</body>
</html>

The SDK is NOT needed here. When loaded through the platform wrapper (/projects/{slug}/), the backend automatically injects the SDK. This HTML is for standalone use only — open it directly in a browser and the project runs independently.

The platform wrapper auto-injects: header, footer, nav strip, and window.Platform. Your main.js can use window.Platform directly without including anything.

4 (Optional) ES Modules & Import Maps

If your project uses ES modules (import statements, like Three.js), add these fields to project.json:

{
  "module": true,
  "importmap": {
    "imports": {
      "three": "https://cdn.jsdelivr.net/npm/three@0.170/build/three.module.js",
      "three/addons/": "https://cdn.jsdelivr.net/npm/three@0.170/examples/jsm/"
    }
  }
}

The platform wrapper will then add <script type="importmap"> and type="module" to your main.js tag automatically. Your project's standalone index.html still needs its own importmap for non-platform use.

5 Write the game logic

projects/my-minigame/main.js

document.addEventListener('DOMContentLoaded', function () {
  var root = document.getElementById('project-root')
  if (!root) return

  // Your game UI here
  root.innerHTML = '<h2 style="color:var(--accent);">My Minigame</h2>'

  // ── Using the Platform SDK (optional) ──
  var p = window.Platform

  // Save score
  if (p && p.auth.isLoggedIn()) {
    p.scores.submit('my-minigame', 100, { level: 3 })
      .then(function () { console.log('Score saved!') })
      .catch(function () { console.log('Score failed') })
  }

  // Load leaderboard
  if (p) {
    p.scores.leaderboard('my-minigame')
      .then(function (scores) { console.log('Leaderboard:', scores) })
  }
})

How Auto-Discovery Works

When the backend starts (or when a request hits /api/platform/projects), it scans the projects/ directory for subfolders containing a project.json file. Each valid folder becomes a project.

The listing page at /projects/ and the nav strip on project pages both fetch from this API, so they always show the current set of projects.

To add a project: just create its folder. To remove one: delete the folder. No build, no config changes.

How It Runs

How to Run

The project is backend-driven — the Python server serves everything (API, frontend pages, project wrappers, docs). No package manager required.

🖥️ Run the backend (no npm needed)

cd backend
.venv\Scripts\uvicorn src.main:app --reload --port 8001

# Or use the launcher script:
scripts\dev.ps1

# Then visit: http://localhost:8001/

Everything works from one port: home page, projects, 3D visualizer, docs, API, SDK — all of it.

🔄 (Optional) Frontend dev server for page editing

If you're editing the Astro pages and need hot-reload, use Node.js from frontend/ (requires the frontend dependencies installed first). Keep the backend on port 8001 running for everything else.

Wrapper Architecture (How Projects Load)

When you visit /projects/my-game/, the backend does not serve a static file. Instead, it generates the HTML dynamically at request time. Here's exactly what happens:

  1. Route match: FastAPI's projects_display.py catches /projects/{slug}/
  2. Validate: Checks that projects/{slug}/ exists on disk
  3. Read metadata: Loads project.json for title, description, icon, and module config
  4. Generate HTML: Produces a complete HTML page containing:
    • The platform header (logo + navigation linking to Projects, About, Docs)
    • A <div id="project-root"> where the project mounts
    • The /platform/sdk.js script (auto-injected)
    • A nav strip populated by /api/platform/projects
    • The project's main.js loaded at the bottom
    • The platform footer with health indicator
  5. Module support: If project.json has "module": true, the wrapper adds <script type="importmap"> and loads main.js with type="module"

The wrapper lives entirely in backend/src/routers/projects_display.py (235 lines). It uses Python f-strings for template generation — no Jinja2, no external dependencies.

Key insight: The wrapper never reads the project's index.html. The index.html is only for standalone use when opening the folder directly. For platform use, the wrapper generates everything from scratch, ensuring consistent chrome across all projects.

Platform SDK — How It Works

The SDK is a 38-line IIFE (Immediately Invoked Function Expression) at platform/sdk.js. It has no dependencies, no side effects, and no DOM manipulation. It simply defines window.Platform and exits.

Why It Works

The SDK works because of same-origin sharing. The project's main.js runs on the same domain as the platform (both served by the same backend). This means:

  • localStorage is shared. When the user logs in via the platform, the JWT token is stored in localStorage. The project's main.js can read it directly via window.Platform.auth.getToken()
  • Fetch calls are same-origin. API calls like fetch('/api/scores') work automatically because they're on the same domain — no CORS headers needed
  • No iframes. The project runs directly on the page, not in an iframe. This means CSS, fonts, and event listeners all share the same context

Internal Structure

window.Platform {
  auth: {
    isLoggedIn()  → reads localStorage.getItem('token')
    getUserName() → reads localStorage.getItem('user_name')
    getToken()    → returns raw JWT
    signup()      → POST /api/auth/signup, stores token
    login()       → POST /api/auth/login, stores token
    logout()      → removes items from localStorage
  },
  saves: {
    save(slug, label, state) → POST /api/saves
    load(slug)               → GET /api/saves/{slug}
    deleteSave(id)           → DELETE /api/saves/{id}
  },
  scores: {
    submit(slug, score, meta) → POST /api/scores
    leaderboard(slug)         → GET /api/scores/{slug}
  },
  models: {
    upload(slug, file)        → POST /api/models/upload (multipart)
    list()                    → GET /api/models
    deleteModel(id)           → DELETE /api/models/{id}
  }
}

How to Use It in a Project

The SDK is automatically available when your project runs through the platform wrapper. Your main.js simply checks if it exists and uses it:

// In your project's main.js — no imports needed
var p = window.Platform

// Auth check
if (p && p.auth.isLoggedIn()) {
  console.log('User: ' + p.auth.getUserName())

  // Save game state
  p.saves.save('my-game', 'level-5', { score: 2500, health: 80 })

  // Submit score
  p.scores.submit('my-game', 2500, { level: 5 })
}

// Load leaderboard (public endpoint, no auth needed)
if (p) {
  p.scores.leaderboard('my-game').then(function(scores) {
    scores.forEach(function(s) { console.log(s.name + ': ' + s.score) })
  })
}
The SDK is optional. If your project doesn't call window.Platform, the object sits unused (~2KB in memory). No errors, no side effects, no performance impact. Projects that don't need auth or data persistence work perfectly without any SDK interaction.

When the SDK Is Not Available

If you open the project's index.html directly as a local file (not through the platform), window.Platform may be undefined depending on whether you included the SDK script. The wrapper always injects it, but the standalone HTML does not — so your main.js should always check if (window.Platform) before using it. This is already done in the example above.
