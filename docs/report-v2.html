1. What Changed

Before (v1.0)

Projects were Astro pages. Each project needed a .astro file, a .js file in public/js/projects/, and an entry in projects.ts. Adding a project required editing 3 files + building the frontend.

backend/app/ — flat structure with SQL mixed into route handlers

Frontend (src/ + public/) at project root

After (v2.0)

Projects are self-contained folders in projects/{slug}/. Each works standalone. The platform discovers them automatically — just create a folder.

backend/src/ — Clean Architecture: models, services, repositories separated

Frontend moved to frontend/ with layered structure

3. Improvements

Architecture

  • Clean Architecture layers: models/ (schemas) → services/ (business logic) → repositories/ (database) → routers/ (HTTP)
  • Dependency direction: Outer layers import inner layers, never the reverse. Routers import services, services import repositories.
  • Thin routers: 60-70% less code per router. Average router: 15 lines vs 45 lines before.
  • Dynamic project system: No build step to add a project. Create a folder → it appears.

Security

  • .env gitignored: No more tracked secrets in the repository
  • config/.env.example committed as a template (all values empty)
  • Rate limiting: Built-in per-IP rate limiter (60 req/min)
  • Global error handler: Catches unhandled exceptions, returns consistent JSON

Developer Experience

  • Single generated/ folder: All build artifacts, cache, and temp files in one place. Clean git status.
  • Platform SDK: window.Platform — 40 lines, zero dependencies, works with any project
  • Auto-discovery API: GET /api/platform/projects returns all discovered projects
  • Dynamic wrapper pages: Backend generates the project page with chrome + SDK on-the-fly
  • Standalone project HTML: Every project works when opened directly in a browser
  • ES Module support: project.json supports "module": true and "importmap" for Three.js and other module-based projects
  • Full-height pages: Both listing and wrapper pages use flex: 1 to fill the viewport
  • SDK auto-injected: The wrapper page adds platform/sdk.js automatically — projects don't need to include it
  • Docs at /docs/: Documentation and migration report moved to root docs/ folder, served at /docs/ and /docs/report-v2.html

Performance

  • Eliminated N+1 queries: JWT now contains user_id, removing subquery on every authenticated request
  • Dead code removed: Orphaned schemas, unused imports, stale comments
  • Project pages compiled to static: Frontend pages (home, about) are pre-built HTML

5. How the New Project System Works

projects/hello-world/
├── project.json      ← Metadata: title, description, icon, category
├── index.html        ← Standalone HTML (works if opened directly)
└── main.js           ← Game logic (loaded by platform wrapper)

For ES module projects (Three.js), project.json can also include:
├── "module": true    ← Tells wrapper to use type="module"
└── "importmap": {...} ← Adds importmap for CDN module resolution

When you visit /projects/hello-world/:

  1. The backend's projects_display.py router matches the URL
  2. It reads projects/hello-world/project.json for metadata
  3. It generates an HTML page with: header + project div + SDK + nav strip + footer + main.js
  4. The project's main.js runs inside the platform, with window.Platform available for API access

The listing page at /projects/ fetches from /api/platform/projects at runtime, so it always reflects the current set of project folders.
